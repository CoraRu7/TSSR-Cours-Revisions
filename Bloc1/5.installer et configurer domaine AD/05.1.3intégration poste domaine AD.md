# Fiche 1 – Concepts clés : Intégration d’un poste client au domaine
</summary>

## Objectif
Apprendre à intégrer un poste client Windows dans un domaine Active Directory pour centraliser l’authentification et la gestion des utilisateurs.

## Définitions
- **Active Directory (AD)** : base de données et ensemble de services qui gère utilisateurs, ordinateurs et ressources réseau.  
- **Domaine AD** : regroupement logique de machines et d’utilisateurs gérés par un contrôleur de domaine.  
- **Poste client** : ordinateur connecté au domaine et authentifié via AD.  
- **Samba AD** : équivalent open-source d’Active Directory pour Linux, offrant services AD, DNS, Kerberos, LDAP, SMB et GPO.

## Importance
- Authentifie et identifie les postes clients.  
- Permet la centralisation des droits, accès aux ressources et sécurité réseau.

# Fiche 2 – Étapes d’intégration d’un poste client dans AD
</summary>

1. **Accéder aux propriétés du système**  
   - Clic-droit sur « Ce PC » → Propriétés  
   - Ou Panneau de configuration → Système et sécurité → Système  

2. **Paramètres système avancés**  
   - Cliquer sur « Protection du système » ou « Paramètres système avancés ».  

3. **Modification du nom de l’ordinateur et du domaine**  
   - Onglet « Nom de l’ordinateur » → Modifier  
   - Dans « Membre d’un », choisir « Domaine »  
   - Saisir le nom du domaine, ex. `SRV.STUDI` → OK  

4. **Authentification du compte**  
   - Entrer un compte avec privilèges pour joindre le domaine : `SRV\dam` + mot de passe → OK  

5. **Validation**  
   - Message : « Bienvenue dans le domaine SRV.STUDI »  

6. **Redémarrage**  
   - Redémarrer le poste pour appliquer les modifications  

7. **Vérification**  
   - À l’écran de connexion, le domaine du poste doit être affiché

# Fiche 3 – Samba Active Directory (open-source)
</summary>

## Historique et fonctionnalités
- Samba AD : créé pour Linux, basé sur le protocole SMB (Server Message Block)  
- Gestion centralisée : authentification, groupes, droits, fichiers et imprimantes  
- Version 4 (2005-2012) : contrôleur de domaine complet, inclut DNS, LDAP, Kerberos, RPC, GPO  
- Fonctionnalités supplémentaires :  
  - Export des GPO en XML pour sauvegarde  
  - Sauvegarde hors ligne sécurisée  
  - Compatibilité Python 2 et 3  
  - Consultation des ID d’événements Windows et noms d’utilisateur  

## Avantages
- Pas de frais de licence  
- Compatible avec infrastructure Linux  
- Remplace ou complète un Active Directory Microsoft

# Fiche 4 – Bonnes pratiques pour les postes et DC virtualisés
</summary>

1. **Contrôleur de domaine virtualisé**  
   - À partir de Windows Server 2012 : support de VM Generation-ID  
   - Configurez le démarrage automatique en 0 secondes  

2. **Points de contrôle**  
   - Éviter les snapshots (points de contrôle) sur un DC pour prévenir les conflits d’horloge et de SID  

3. **Synchronisation de l’heure**  
   - Désactiver la synchronisation automatique avec l’hôte pour un DC virtualisé  
   - Le DC doit être maître de l’heure locale  

4. **Ne pas restaurer un DC depuis un état enregistré**  
   - Risque de corruption de l’horloge et de l’AD  

5. **Conversion ou mise à niveau**  
   - Ne pas convertir DC physique/virtuel  
   - Mise à niveau sur place non supportée : déployer un nouveau serveur et migrer les rôles FSMO  

6. **Réplicas et site distant**  
   - Préférer un autre DC pour la réplication AD dans un site distant  
   - Réplicas rarement recommandés sauf cas de DRP

# Fiche 5 – Résumé pratique
</summary>

- **Objectif principal** : intégrer un poste client dans le domaine pour centraliser la gestion et l’authentification.  
- **Processus clé** : Propriétés du système → Modifier nom/domaine → Authentification → Redémarrage → Vérification.  
- **Samba 4** : alternative open-source complète, compatible Linux, gestion centralisée et GPO.  
- **Sécurité VM** : respecter les règles pour les DC virtualisés (pas de snapshot, pas de conversion, maître de l’heure, migration FSMO planifiée).
